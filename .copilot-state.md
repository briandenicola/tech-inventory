## Key debugging breakthroughs this session
1. **v1 vs v2 token issuer**: Brian's Entra app registration's `accessTokenAcceptedVersion=null/1` emits `sts.windows.net/<tenant>/` issuer. MSAL.js requests v2 by default but issuer follows the manifest, not the request. JwtBearer's metadata-derived issuer is v2 only — must set `ValidIssuers` explicitly with both forms.
2. **Entra `roles` claim parsing**: `JwtSecurityTokenHandler` expands JSON array claims into ONE Claim per array element, each with bare string Value (NOT JSON-encoded). Original `JsonSerializer.Deserialize<string[]>(rolesClaim.Value)` threw on every Entra token.
3. **`MapInboundClaims=true` (default)**: JwtBearer rewrites short JWT claims (`oid`, `sub`, `name`, `roles`) to SOAP-era URIs from `DefaultInboundClaimTypeMap` before the ClaimsPrincipal is built. Any code looking up `FindFirst("oid")` returns null. Fix: check both the short name and the mapped URI.
4. **App role naming convention**: Brian renamed Entra App role values from `App.View`/`App.Write`/`App.All` to `Viewer`/`Member`/`Admin` to match backend `RequireRole("Admin")` policy (30-second Entra edit vs translation layer forever).
5. **Service worker "401 from service worker" was a red herring**: PWA's StaleWhileRevalidate on GET `/api/v1/*` doesn't cache 401s (only `[0, 200]`). The 401 was a real network response — controller's `GetCurrentOwner` returning `Unauthorized()` after `Guid.TryParse` failed on the v1 `sub` hash.
