## Problem
Brian asked two related questions during the 2026-05-19 PWA field-test:
1. "Can we have an exportable report for insurance purposes?"
2. "What happened to our reporting backlog that was defined?"

**Answer to (2)**: Reporting was scoped into what the **original** PRD §13
called "Phase 4 — Polish & Reporting" — but that label predates the
2026-05-19 phase-model rewrite. Under the current model (PRD §13, rewritten
2026-05-19), all post-MVP reporting work lives in **P4 — Continuous
Iteration** and is tracked as discrete `F0XX-*.md` backlog entries.
Reporting has lived only in PRD/plan prose until this file; **F032 is the
first concrete reporting backlog entry**. Sources:
- `specs/002-frontend-mvp/plan.md:238` — "To Phase 4 (Polish & Reporting):
  Dashboard, Timeline View, Saved Views, Advanced Search." (Historical
  reference; that "Phase 4" maps to P4 Continuous Iteration today.)
- `docs/prd.md` §F4 — **F4. Export** — CSV/JSON, filtered or full, includes
  audit metadata optionally.
- `docs/prd.md` §13 P4 — Reporting is called out as a current Phase-4 theme.
- `docs/prd.md` §U10 — *"As Admin, I can create a full export of all items
  in the household."*

**Answer to (1)** is the deliverable in this entry: ship the first concrete
piece of Phase-4 reporting as an insurance-grade export. The
generic CSV/JSON export from PRD F4 doesn't satisfy an insurance use case
on its own — Brian needs a printable, dated, totals-bearing document he can
hand an insurance agent after a flood or theft event.

## Proposed Solution
Add an "Insurance Report" export option in addition to the planned CSV/JSON
exports. Produce a single PDF (or print-styled HTML) that includes:

- Cover sheet: household identifier, report timestamp, total device count,
  total declared replacement value (sum of `Device.PurchasePrice` if set,
  with "value unknown" line item count).
- Per-room (Location) sections: each device with brand, model, serial
  number, purchase date, purchase price, current status, owner.
- Photos column (when image-upload from F033 ships): thumbnail per device.
  Until then: blank placeholder column.
- Footer on every page: report ID + page N of M + "Generated by Tech
  Inventory v{version}, source data as of {timestamp}".

### Export pipeline
- Reuse the existing `Exports` application slice (`src/TechInventory.
  Application/Exports`) — add a new exporter alongside CSV/JSON.
- PDF generation choice: lean toward print-styled HTML rendered server-side
  (Razor view or templated Svelte) → headless Chromium via a deterministic
  container step. Requires an ADR (Ripley) — open question below.
- Export request is async if dataset > N rows: `POST
  /api/v1/exports/insurance-report` returns 202 + status URL; client polls.
  Mirror the F4 export pattern.

### Frontend
- New "Export → Insurance Report (PDF)" entry in the device list export menu
  and on a `/admin/reports` page (the start of the Phase-4 reports hub).
- Filter at export time: include only `Active` devices by default; user can
  override.
- Download lands as `inventory-insurance-{yyyy-MM-dd}.pdf`.

## User Stories
- *As Admin, I generate an insurance-ready PDF in under 30 seconds for a
  household of ~300 devices.*
- *As Admin, I can include or exclude retired/disposed devices from the
  insurance report.*
- *As Admin, I have a one-page cover with totals I can hand to an insurance
  agent without further editing.*

## Acceptance Criteria

      unit tests covering: per-location grouping, totals, unknown-price
      handling, empty result
- [ ] `POST /api/v1/exports/insurance-report` accepts the same filter shape
      as `/api/v1/devices`; behind `Admin` policy

      pixel-exact)
- [ ] Frontend "Insurance Report (PDF)" entry under Export menu on

      statuses
- [ ] Filename pattern matches `inventory-insurance-{yyyy-MM-dd}.pdf`

## Out of Scope
- Dashboard widgets, charts, "warranty expiring soon" — separate Phase-4
  backlog entries to come.
- Replacement-value estimation (current → today's price). Defer; v1.2
  reports declared purchase price only.
- Email-the-report-to-myself. Defer; user downloads and forwards.
- Per-room photos in the PDF until F033 ships image upload.

## Dependencies
- Ripley: ADR for PDF generation strategy (headless Chromium vs. server-
  side library like QuestPDF). Material decision; locks a dependency.
- Hudson: container sizing if we add a headless Chromium step.
- F033 (image upload) is *not* a blocker — placeholders ship without it.
- F031 (merge reference data) is not a blocker but will produce cleaner
  reports — call this out in the user-facing docs once both ship.

## Open Questions
- **PDF library choice.** Headless Chromium gives pixel-perfect HTML→PDF
  but adds a heavy container dependency. QuestPDF (MIT C# library) avoids
  the dependency but means a second template pipeline. Ripley to decide
  via ADR.
- Do we sign or notarize the PDF for insurance evidentiary use? Out of
  scope for v1.2 but worth noting; insurance agents typically accept a
  plain PDF with date/totals.
- Should the report include the audit-log trail per device? Useful for
  proving custody; default off, opt-in flag in the export request.
